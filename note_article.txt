HiDock P1の公式アプリがひどすぎたので、USBプロトコルを解析してCLIツールを作った

きっかけ

HiDock P1というAI録音機器を使っている。会議や思考の整理に使っていて、音質も機能も悪くない。問題は録音データを取り出すための公式アプリ「HiNotes」だ。

動作は重く、UIは直感的でなく、ファイルを取り出すだけなのに毎回アプリを起動して手動でポチポチする必要がある。録音機器として優秀なのに、データへのアクセスがボトルネックになっていた。

「何かいい方法はないか」と思ったのが始まりだった。

まずデバイスを調べた

HiDock P1をMacに繋いでも、普通のUSBドライブとしては認識されない。公式の説明によると「ディスクは保護されている」とのことで、HiNotesアプリ経由でしかアクセスできない設計になっている。

ならばHiNotesが内部でどうやってデバイスと通信しているか調べればいい。USBの通信をキャプチャして解析した結果、以下のことがわかった。

- Vendor ID: 0x10D6、Product ID: 0xB00E
- インターフェース: Class 0xFF（ベンダー独自）
- 内部プロトコル名: Jensen

Jensenプロトコルの構造

パケットは12バイトのヘッダ＋ボディという構成だった。

[0x12][0x34][CMD_HI][CMD_LO][SEQ x4][LEN x4][BODY...]

主要なコマンドは3つで十分だった。

CMD     用途
0x0004  ファイル一覧取得（名前・サイズ・MD5ハッシュ）
0x0005  ファイルのダウンロード
0x0006  ファイル数取得

ファイル一覧を取得してみると、2026Jan01-163234-Wip00.hda のようなファイル名が並んで返ってきた。拡張子は .hda だが、中身をhexダンプすると先頭が FF FB——MP3のマジックバイトそのものだった。ただ拡張子を変えているだけだった。

Pythonで実装した

pyusb を使って同じプロトコルをPythonで実装した。構成はシンプルにした。

hidock_reader/
├── device.py    # USB通信・Jensenプロトコル
└── transfer.py  # 差分コピー・.hda→.mp3リネーム

実行は一行で済む。

sudo python3 -m hidock_reader --dest ~/Recordings

これだけで、まだコピーしていないファイルだけを自動で取り出して .mp3 として保存する。既存ファイルはスキップされる。--days 7 オプションをつければ直近7日分だけに絞ることもできる。

UIよりコマンドライン一発の方が便利だった

作ってみて気づいたのは、UIより自動化との親和性が高いということだ。

たとえば macOS の launchd を使えば、USBを挿した瞬間にこのスクリプトが自動起動して録音データをコピーする仕組みも作れる。GUIアプリでは絶対にできないことだ。

私のワークフローはこうなっている。

1. HiDock P1をUSBで繋ぐ
2. このツールで録音ファイルを指定フォルダにコピー
3. そのフォルダを1分ごとに監視していて、新しいファイルが来たら自動でWhisperにかけてテキスト化
4. 1週間分の文字起こしがまとめて閲覧できるようになっている

録音→テキスト化→活動記録の閲覧まで、ほぼ全自動になった。

OSSとして公開した

同じ悩みを持つ人のために公開した。

https://github.com/sahiruha/HiDock_extracter

pyusb と libusb があれば動く。macOS・Linuxで動作確認済み。

公式アプリのUIが気に入らないなら、プロトコルを解析して自分で作ればいい——というのが今回の結論だった。
